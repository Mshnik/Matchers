# Matchers: Type-Safe, Declarative Value Matching

[![Language](https://img.shields.io/badge/language-Kotlin-purple.svg)](https://kotlinlang.org/)
[![License](https://img.shields.io/badge/license-Apache%202.0-blue.svg)](https://www.apache.org/licenses/LICENSE-2.0.txt)

`matchers` is a lightweight, expressive, and strictly type-safe
evaluation engine built on top of Protocol Buffers (protobuf). It decouples
business validation logic from application runtimes by serializing match
evaluation graphs into pure Proto bytes, and provides a fluent Kotlin DSL to
write runtime-safe matching rules.

---

## 1. Quick Start / Basic Usage

Define complex matching conditions fluently using the type-safe DSL, compile the
rule once, and instantly run evaluations against your data classes or raw
Protobuf payloads.

```kotlin

import com.redpup.matchers.testing.proto.TestMessage


// Define a type-locked evaluation graph.
val rule: Matcher = matcher {
  messageMatcher(TestMessage.getDescriptor()) {
    "string_value" value "active"
    "int32_value".matches<Int> {
      not { value(0) }
    }
    "string_values" matchesCollection {
      containsDistinct {
        matches { startsWith("prefix_") }
        matches { inSet(setOf("match_a", "match_b")) }
      }
    }
  }
}

// Compile into an immutable engine instance
val engine: KMatcher<TestMessage> = KMatcher.compile(rule)

// Evaluate dynamically at runtime
val message: TestMessage = fetchIncomingMessage()
val isMatch: Boolean = engine.matchTyped(message) // Returns true or false
````

## 2. Motivation

Traditional rule evaluation requires deploying new code, writing brittle
boilerplate reflection, or parsing untyped JSON structures at runtime.

This library solves these hurdles by treating **the rule evaluation itself as a
serializable schema**:

* **Serializable Business Logic:** Because the rules are defined in standard
  Protobuf (`Matcher`), rules can be stored in a database, written in config
  files, streamed over gRPC networks, or dynamically generated by a frontend UI.
* **Compile-Time Safety (Phantom Types):** Standard protobuf builders allow you
  to accidentally insert a `String` constraint inside an `Int32` property block.
  This library implements a phantom type-bounding
  layer (`TypedMatcherBuilder<T>`) that forces structural compliance at
  compilation time.
* **Zero Overhead Deserialization:** The evaluation graph maps natively to the
  underlying fields, moving all intermediate object allocations and parsing
  to a dedicated compilation step and keeping your hot path clear.

## 3. Construction & The Kotlin DSL

The DSL exposes expressive extension hooks to create match constraints safely.

### Value and Set Testing

Test explicit bounds or membership cleanly. The library natively
overloads `anyOf` with arrays/varargs to bypass list boilerplate:

```kotlin
typedMatcher<Int> { value(42) }
typedMatcher<Int> { inSet(listOf(10, 20, 30)) }
typedMatcher<String> {
  anyOf(
    "admin",
    "moderator",
    "owner"
  )
}
```

### Relational Comparisons

Evaluate relational inequalities against any primitive type or custom data type
implementing Comparable<T> (such as Int, String, or custom timestamp classes).
Enums are automatically supported by routing comparisons through their
underlying numeric values:

```kotlin
// Matches values strictly greater than 100
typedMatcher { greaterThan(100) }

// Lexicographical string bounds checks
typedMatcher { lessThanOrEqual("beta") }

// Structural enum index threshold evaluations
typedMatcher { greaterThan(TestEnum.TEST_ENUM_1) }
```

### String Strategies

Leverage semantic string processing with optional case-sensitivity rules:

```kotlin
typedMatcher<String> { textEquals("hello", CaseSensitivity.CASE_INSENSITIVE) }
typedMatcher<String> { startsWith("PROD_") }
typedMatcher<String> { matchesRegex("^[a-zA-Z0-9]+$") }
```

### Message Graph Routing

Traverse nested objects using either explicit `FieldDescriptor` references or
string names mapped seamlessly through runtime descriptor schemas:

```kotlin
untypedMessageMatcher(TestMessage.getDescriptor()) {
  "message_value" matchesMessage {
    "bool_value" value true
  }
}
```

### Combiners and Inversion

Build arbitrary nested boolean logic trees using clear block structuring:

```kotlin
typedMatcher<Int> {
  allOf {
    matches { not { value(0) } }
    matches { anyOf(10, 20, 30) }
  }
}
```

## 4. Compilation via `KMatcher.compile`

To maintain performance, evaluation graphs must not be re-interpreted every time
an item is evaluated. The library isolates parsing and graph building from
execution via `KMatcher.compile`:

```kotlin
// Build once or pull raw from external transport bytes
val rawProto: Matcher = Matcher.parseFrom(inputStream)

// Compilation builds an indexing lookup tree for O(1) field checks
val matcherEngine = KMatcher.compile<TestMessage>(rawProto)
```

> **Architecture Tip:** Always parse and invoke `KMatcher.compile` during
> application startup or rule-update events, keeping it completely outside of
> hot request processing loops.

---

## 5. Runtime Matching Execution

Once compiled, you can evaluate payloads in two ways depending on context safety
requirements.

### Strongly-Typed Evaluation (Recommended)

Guarantees type alignment using implicit Kotlin signatures:

```kotlin
val result: Boolean = matcherEngine.matchTyped(testMessageInstance)
```

### Untyped Dynamic Evaluation

Allows checking generic payloads, generic message interfaces, or polymorphic
data blocks when the underlying types cannot be verified at compile time:

```kotlin
val dynamicMessage: Any = getDynamicPayload()
val result: Boolean = matcherEngine.match(dynamicMessage)
```

## 6. Advanced Operations: Transformations & Conversions

### Reverting to Proto Value Shapes

Every rule generated by the custom domain DSL can be unwrapped directly back
into its underlying data payload configuration for transport or database
storage:

```kotlin
val myDslMatcher = typedMatcher<String> { startsWith("A") }

// Extract the exact underlying data shape
val protoPayload: Matcher = myDslMatcher.delegate.build()
val serializedBytes: ByteArray = protoPayload.toByteArray()
```

## 7. Java Interoperability

While designed primarily for idiomatic Kotlin, `redpup-matchers` provides full
interoperability with enterprise Java codebases.

### Avoid Inlined Extension Functions in Pure Java

Kotlin `inline` methods with `crossinline` lambda blocks compile into
specialized bytecode signatures that can be verbose and difficult to write
directly in raw Java.

### Java Integration Best Practices

When interacting with the framework from Java, construct matchers using standard
generated Java protobuf builders, then pass them to `KMatcher` for compilation
and evaluation:

```java
import com.redpup.matchers.KMatcher;
import com.redpup.matchers.proto.Matcher;
import com.redpup.matchers.proto.ValueMatcher;
import com.redpup.matchers.testing.proto.TestMessage;

public class ComplianceChecker {

  private final KMatcher<TestMessage> engine;

  public ComplianceChecker(byte[] ruleBytes) throws Exception {
    // Construct using native Java Proto builders
    Matcher rule = Matcher.newBuilder()
        .setValueMatcher(ValueMatcher.newBuilder().setInt32Value(42))
        .build();

    // Compile and use seamlessly
    this.engine = KMatcher.Companion.compile(rule);
  }

  public boolean evaluate(TestMessage message) {
    return this.engine.matchTyped(message);
  }
}
```

## 8. Performance Implications & Benchmarks

The runtime evaluation behavior scales predictably based on the type of matcher
blocks applied.

### Algorithmic Complexities

| Matcher Type                              | Time Complexity  | Notes / Allocation Profile                                                        |
|:------------------------------------------|:-----------------|:----------------------------------------------------------------------------------|
| `ValueMatcher`                            | $O(1)$           | Direct primitive equality check. No heap allocations.                             |
| `ComparisonMatcher`                       | $O(1)$           | Direct numeric/lexicographical comparison. No heap allocations.                   |
| `ValueInSetMatcher`                       | $O(1)$           | Backed internally by optimized hashed lookup sets.                                |
| `StringMatcher` (Regex)                   | Varies           | Generally linear to string length ($N$), but can grow with pattern complexity.    |
| `MessageMatcher`                          | $O(F)$           | Linear to number of constraints ($F$) evaluated. Short-circuits on first failure. |
| `CombiningMatcher`                        | $O(M)$           | Evaluates $M$ matchers. Short-circuits on first failure.                          |
| `CollectionMatcher` (`contains_elements`) | $O(M \cdot E^M)$ | Modeled as a bipartite matching assignment problem. Uses backtracking.            |

### Distinct Elements Collection Strategy Warnings

The `contains_elements` query runs a backtracking search engine to solve
structural uniqueness across collections. While highly powerful for structural
validation, it has a worst-case exponential time cost of $O(M \cdot E^M)$ (where
$M$ is the filter count and $E$ is collection size). Keep lists and sub-matcher
counts reasonably constrained within performance-critical, low-latency execution
paths.
